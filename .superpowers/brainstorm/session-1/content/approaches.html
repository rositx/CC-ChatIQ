Design Options for Phase 3 Completion

We need to complete the agent-to-agent transfer mechanism and optimize token streaming rendering performance. Select the design approach to proceed with.

Option A: Real-Time Socket-Driven Transfer & Scoped Selectors (Recommended)

1. Live Roster & WebSocket Routing

Maintain agent presence in Redis with heartbeats. Transfer endpoint POST /api/v1/sessions/{session_id}/transfer updates the session in the DB and immediately broadcasts a session_transfer WebSocket frame. The target agent's UI updates instantly without polling.

2. Granular Memoization

Remove token length hooks from the parent stream component to prevent it from re-rendering on every token. Wrap message bubbles in React.memo and use fine-grained selectors so only the active streaming bubble re-renders while the rest of the list remains untouched.

Pros

  • Sub-50ms transfer UI updates
  • Zero re-renders on historical chat list
  • No network polling overhead

Cons

  • More complex state management in the frontend store
Option B: Polling-Based Transfer & Debounced State Streams

1. REST-Based Presence & Polling

Keep agents simple: no Redis presence hashes. Dashboard UI polls active session queues every 3-5 seconds. Transfers update the database, and other agents see the transferred session on their next polling interval.

2. Debounced Token Appends

Instead of rendering tokens character-by-character, debounce the socket messages on the client side, pushing batched chunks of 5-10 tokens to the state. This reduces the frequency of re-renders but makes the stream look jerky.

Pros

  • Easier to implement
  • Simple backend endpoints

Cons

  • Jerky/laggy text streaming
  • Increased database load due to constant polling
  • No instant push alerts for manual transfers
